# Project Health Assessment Methodology

## Objective

The Project Health Reporting Agent determines project health using a deterministic, explainable scoring framework. The objective is to provide consistent and auditable Green / Amber / Red (RAG) classifications while minimizing subjectivity and ensuring that health assessments can be traced back to measurable project indicators.

---

## Design Philosophy

A key design decision in this system is the separation of project assessment from narrative generation.

```text
Project Signals
      ↓
Deterministic Scoring Engine
      ↓
Green / Amber / Red

LLM
      ↓
Explanation Only
```

The Large Language Model (LLM) is **not responsible for determining project health**. Instead, it is used solely to explain the results generated by the scoring engine. This ensures consistency, transparency, and repeatability across assessments.

---

## Health Indicators

The scoring framework evaluates multiple dimensions of project execution:

| Indicator | Purpose |
|------------|-----------|
| Schedule Slippage | Measures deviation from expected project progress |
| Variance Health | Evaluates average task schedule variance |
| Overdue Task Ratio | Measures proportion of tasks overdue relative to plan |
| Milestone Health | Assesses milestone completion performance |
| Blocker Density | Measures concentration of critical blockers |
| Schedule Health Signal | Uses schedule health metadata available in source files |
| Comment Risk Signal | Extracts delivery risk indicators from stakeholder comments |

Each indicator is normalized into a common scoring scale before aggregation.

---

## Composite Health Score

The overall project score is calculated using weighted project signals.

```text
Composite Score
    =
Weighted Sum of Available Signals
```

The framework dynamically adjusts for missing information by considering only available indicators and tracking score coverage.

This prevents projects with incomplete data from being unfairly penalized while still reflecting reduced confidence in the final assessment.

---

## Override Rules

Certain project conditions represent material delivery risks and should not be masked by favorable scores in other areas.

The system therefore applies deterministic override rules:

### Rule 1: Critical Overdue Activities

Projects containing overdue critical-path activities cannot be classified as Green.

### Rule 2: Excessive Blockers

Projects with blocker density above defined thresholds are automatically escalated to Red.

### Rule 3: High Overdue Task Concentration

Projects with a large proportion of overdue active tasks are automatically escalated to Red.

These overrides ensure that severe execution risks are surfaced even when other indicators appear healthy.

---

## Confidence Assessment

In addition to project health, the framework produces a confidence score.

Confidence is influenced by:

- Availability of project data
- Coverage of scoring signals
- Completeness of schedule information
- Availability of comments and risk indicators

Higher confidence indicates stronger evidence supporting the final classification.

---

## Data Quality Assessment

Project plans frequently contain incomplete or inconsistent information.

The system evaluates:

- Missing status values
- Missing schedule health fields
- Missing dates
- Missing critical-path indicators

A Data Quality Score is generated and included alongside every assessment.

This allows stakeholders to distinguish between:

- A healthy project with reliable data
- A healthy project with poor data quality

---

## RAG Classification

The final project classification is determined from the composite score and override rules.

| Classification | Interpretation |
|----------------|----------------|
| Green | Project is progressing within acceptable limits |
| Amber | Emerging concerns require monitoring or corrective action |
| Red | Significant delivery risks require immediate attention |

The resulting classification is accompanied by supporting evidence, confidence metrics, and executive-level explanations.

---

## Explainability

Every RAG assessment can be traced back to:

- Individual project indicators
- Applied scoring weights
- Override rules
- Data quality metrics
- Extracted risk signals

This ensures that project health decisions remain transparent, auditable, and suitable for enterprise reporting environments.
